==================================================
　どうやって動くのか？（How does it all work?）
==================================================


■ 1. プログラムは「レシピ」のようなもの
--------------------------------------------------
・コンピュータはプログラム（＝プログラマーの指示）に
　書かれたことを、そのまま正確に実行するだけ
・コンピュータには「正確に（EXACTLY）」何をするか伝える必要がある

【レシピのたとえ】
・ケーキを焼くには、決まった順序で作業をこなす必要がある
・例：「卵を4個ボウルに入れて混ぜる」という作業
　- 人間なら「殻を割って中身だけ使う」と当然のように考える
　- しかしコンピュータは、殻ごとボウルに入れて混ぜてしまうかもしれない
・つまりプログラムを書くときは、非常に明確（explicit）に
　指示しないと、結果が間違ったものになる


■ 2. プログラミング言語とソースコード
--------------------------------------------------
・プログラムを書くにはプログラミング言語を使う
・世の中には非常に多くの言語があり、この授業では C++ を使う

▼ ソースコード（source code）
　・私たちが書くコードのこと
　・人間が理解できるように、コンピュータコードより高水準（high-level）
　・人間は長い0と1の羅列を読むのが苦手なので、人間向けに書く


■ 3. ビルドの流れ（登場人物）
--------------------------------------------------

▼ エディタ（Editor）
　・プログラムの文章（テキスト）を入力するためのもの
　・ワープロのような、特定の言語専用に設計されたツール
　・ファイルの拡張子は .cpp / .h（他に .c / .hpp などもある）
　　※このコースでは .cpp と .h を使う

▼ オブジェクトコード（object code）
　・コンピュータが理解できる低水準（low-level）の形式
　・コンピュータはソースコードを直接は読めないので、変換が必要

▼ コンパイラ（Compiler）＝ 翻訳者
　・入力：C++のソースコード → 出力：オブジェクトコード
　・高水準から低水準へ翻訳する
　・ソースコードにエラーがあると、オブジェクトコードは生成されない

▼ リンカ（Linker）
　・自分のオブジェクトコードと、既存のライブラリ
　　（他のプログラマーが書いたコード）をまとめてつなぐ
　・実行可能なプログラム（executable）を作る
　・既存のコードはできるだけ再利用したいので、これはよくある流れ

▼ テストとデバッグ（Testing & Debugging）
　・できるだけ多くのエラーを見つけ（テスト）、
　　それらを取り除く（デバッグ）


==================================================
　C++のビルドプロセス（The C++ Build Process）
==================================================

【流れ図】
　ソースコード         コンパイル        オブジェクトコード      リンク           実行ファイル
　Source Code   →   C++ Compiler  →   Object Code     →   Linker     →   Executable
　file1.cpp                            file1.obj                            main.exe
　file2.cpp                            file2.obj
　file3.cpp                            file3.obj
　main.cpp                             main.obj

・リンク時には次のものも一緒に結合される
　- C++標準ライブラリ（C++ Standard Library）
　- その他のライブラリ（Other Libraries）

【補足（拡張子について）】
・オブジェクトファイル
　- Windows：.obj
　- Mac / UNIX系：.o
・実行ファイル
　- Windows：.exe
　- Mac / UNIX系：通常は拡張子なし

【ファイル数について】
・このコースでは、プログラムが大きくなるまでは
　1つの .cpp ファイルだけを使う
・大きくなると、整理のため複数のソースファイルが必要になる
・実務のC++プログラムは、数千のソースファイル・
　数百万行に及ぶこともある

※ 注意：ここまで終わっても完成ではない。
　 必ずテストで論理エラー（logic errors）を見つけ、
　 デバッグで取り除く必要がある。


==================================================
　統合開発環境（IDEs: Integrated Development Environments）
==================================================

■ IDEとは
--------------------------------------------------
・これまでの作業を1つの画面でまとめて行えるプログラム
・ボタンを押すだけでビルド全体が進むように見えることもある
・IDEに含まれる主な機能
　- エディタ（Editor）
　- コンパイラ（Compiler）
　- リンカ（Linker）
　- デバッガ（Debugger）
　- すべてのファイルを同期させる（Keep everything in sync）
　- バージョン管理システム（Git や Subversion など）との連携

※ 多くのC++コースはデバッガを扱わないが、講師はそれを間違いだと考え、
　 C++の基礎を学んだ後にデバッガを扱う予定。


■ 主なIDE一覧と特徴
--------------------------------------------------

▼ CodeLite ★このコースで使用
　・良いクロスプラットフォームIDE
　・ただしMac版はバグが多く、講師の環境では頻繁にクラッシュした

▼ Code::Blocks 
　・無料、クロスプラットフォーム、高速、メモリ要件が小さい
　・上記の条件をすべて満たすため採用

▼ NetBeans
　・OracleのIDE。主にJava開発用だがC++プラグインがあり、とても良い
　・ただしJava（JRE）のインストールが必要で、メモリ要件が大きい

▼ Eclipse
　・多言語対応、クロスプラットフォーム
　・ただしメモリ要件が大きい

▼ CLion
　・JetBrains社製（Kotlin、IntelliJ、Android Studio を作った会社）
　・非常に優秀で、講師イチオシのIDE
　・ただし無料ではなく、かなり高価（30日間の体験版あり）

▼ Dev-C++
　・Windows専用。小さくて高速、使いやすい

▼ KDevelop
　・クロスプラットフォーム＆多言語対応のIDE
　・ただしMac用バイナリはなく、Macで使うにはソースからビルドが必要

▼ Visual Studio
　・Microsoftの開発環境。C++版はWindows専用
　・Mac版もあるがC++には対応していない

▼ Xcode
　・Appleの開発環境。Mac専用
　・Mac開発者ならiOS/macOS/Swift開発で既に使っている可能性が高い
　・優れた開発環境


■ IDE選びについてのアドバイス
--------------------------------------------------
・完璧なIDEは存在しない
・プロの開発者は、チームで使っているIDEに合わせることが多い
・コンサルタントなら、できるだけ多くのIDEを学ぶとよい
・エディタ＋コマンドラインで作業できる環境も持っておくと便利
・C++に慣れてきたら、いろいろ試して自分に合うものを見つけよう
・良いIDEを見つけたら、コースのフォーラムで共有を
